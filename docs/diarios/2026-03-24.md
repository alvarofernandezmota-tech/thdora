### Decisiones arquitectónicas tomadas

| Decisión | Detalle | Alternativa descartada |
|----------|---------|------------------------|
| **Monorepo** | Todo en un solo repo: core, api, bot, ai | Repos separados por módulo |
| **ABC interfaces** (`AbstractLifeManager`) | Contrato explícito para cada módulo | Duck typing sin interfaz |
| **FastAPI** para la API REST | Async, tipado, autodoc con OpenAPI | Flask (más simple pero sin async nativo) |
| **Capa de módulos independientes** | `core/`, `api/`, `bot/`, `ai/` | Arquitectura monolítica |
| **JSON como primera persistencia** | Simple, sin dependencias | SQLite (más complejidad ahora) |

**Decisión sobre API vs mismo repo:**  
Se valoró si `AppointmentManager` debía ser un servicio externo accesible via API. Se decidió que **todo debe estar en el mismo repo** porque:
- El proyecto es personal, no necesita microservicios todavía
- Añade complejidad sin beneficio real en esta fase
- El código es más fácil de mantener y entender en un único lugar
- Si en el futuro escala, se puede extraer un módulo concreto

### Interfaces y clases documentadas

#### `AbstractLifeManager` (ABC)

| Método | Descripción | Complejidad |
|--------|-------------|-------------|
| `create_appointment(title, date, description)` | Crea una cita con ID único | O(1) |
| `get_appointments(date_filter)` | Lista citas, opcional filtro por fecha | O(n) |
| `log_habit(name, value, date)` | Registra un hábito diario | O(1) |
| `get_habits(name, date_filter)` | Obtiene historial de un hábito | O(n) |
| `get_day_summary(date)` | Resumen completo del día | O(n) |

Todos los métodos tienen docstrings con: `Args`, `Returns`, `Raises`, `Complexity`, `Example`.

#### `MemoryLifeManager` (implementación)
- Almacenamiento en dicts en memoria (sin persistencia)
- Misma interfaz que `AbstractLifeManager`
- Usada para desarrollo y tests
