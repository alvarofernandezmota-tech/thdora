### 2. Decisión de arquitectura: modo Toki
Se descartó crear un sistema de IA separado ("Thea IA").
Decisión: usar el prompt de Groq como motor de personalidad
y enriquecer el contexto con datos reales de la API antes de cada llamada.
Esto se llamó internamente **modo Toki** por analogía con el asistente Toki.

**Principio:** La IA actúa como router de intención + extractor de slots.
Cuando no entiende, devuelve la interfaz del bot (botones), no texto inventado.

### 3. Debate sobre modelos (Gemma / OpenRouter)
- Gemma 3 no está disponible en Groq todavía
- OpenRouter posible en el futuro pero no necesario ahora
- Decisión: quedarse con Llama 3.3 70B en Groq, el problema no era el modelo

### 4. Implementación v0.14.0

**`src/bot/groq_router.py`:**
- `route()` acepta `api_context: Optional[Dict]`
- `_build_chat_system(api_context)` — inyecta citas + hábitos en el prompt
- Intent `desconocido` → `show_menu=True` (no genera texto)
- Degradación elegante: si la API falla, el bot sigue funcionando

**`src/bot/handlers/nlp.py`:**
- `_get_api_context()` — 3 llamadas paralelas con `asyncio.gather`:
  - `get_appointments(hoy)`
  - `get_appointments(mañana)`
  - `get_habits(hoy)`
- Intent `desconocido` → muestra menú + `_kb_start()`
- Fix hora 00:00: pide confirmación antes de crear

## 💡 Decisiones y aprendizajes

- **El problema no era el modelo, era el prompt y el contexto.**
  Llama 3.3 70B es más que suficiente para este caso de uso.
- **asyncio.gather para las 3 llamadas API** — no añade latencia extra.
- **Degradación elegante siempre**: si algo falla, el bot sigue funcionando.
- **La personalidad del bot = el string `_CHAT_SYSTEM_BASE`**.
  No hace falta ningún sistema separado.
- **Documentar en el momento**, no después: el CHANGELOG y el diario
  se escriben mientras se implementa, no al final.
