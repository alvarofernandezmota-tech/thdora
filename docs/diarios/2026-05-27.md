### El "ladrillo legado" en src/core/

El patrón `AbstractLifeManager` → `JsonLifeManager` / `MemoryLifeManager` / `SqliteLifeManager` es el **diseño anterior al FastAPI**. Cuando se migró a la arquitectura actual (FastAPI + SQLAlchemy), este código quedó huérfano.

```
src/core/ (legado — no importado por nadie en src/bot/ ni src/api/)
    interfaces/abstract_lifemanager.py   ← interfaz Strategy del THDORA v1
    impl/json_lifemanager.py             ← backend JSON (descartado)
    impl/memory_lifemanager.py           ← backend en memoria (tests)
    impl/sqlite_lifemanager.py           ← backend SQLite directo (sustituido por FastAPI)
    ocr/                                 ← vacías, idea futura descartada o pendiente
```

**Decisión:** Este código no se borra todavía. Los tests `test_json_lifemanager.py` y
`test_memory_lifemanager.py` lo usan. Se marca como `# LEGACY` y se evaluará en Fase 5.

### Notas sobre src/db/models.py

- `time` en `Appointment` es `nullable=False` → la cita `id:4` sin time **romperá** cualquier
  query que devuelva esa fila. Limpiar en DBeaver es urgente.
- `user_id` en `UserConfig` es `String(50)`, no `BigInteger` — funciona pero Gemini tiene razón:
  los IDs de Telegram son `int64`, mejor `BigInteger` en la migración a Postgres.
- No hay FK entre tablas — diseño consciente (documentado en docstring del modelo).
  Correcto para MVP mono-usuario. Cambiar con multi-usuario.

### src/ai/ — la capa AI está bien separada

```
src/ai/
    groq_client.py    ← wrapper del SDK de Groq (async, timeout, retry)
    intent_parser.py  ← parsea la respuesta JSON de Groq a intent + entidades
```

Esta separación es exactamente lo que Gemini recomiendó para migrar a Tool Calling.
No hay que reestructurar nada — solo modificar `intent_parser.py` para usar `tools=[]`
en la llamada a Groq.
