## 🏗️ Plan de profesionalización — decidido hoy

### Filosofía correcta
> "Primero la base sólida, luego las features"

El error anterior era añadir features (F12 IA, F14 Twilio, F15 Whisper) sin tener la base probada y visible. El nuevo orden es:

```
PASO 1 → Fix bug #10 /config (10 min, 3 líneas)
PASO 2 → Visualizar BBDD en DBeaver (ver los datos reales)
PASO 3 → Verificar bot comando a comando (checklist producción)
PASO 4 → Renovar GROQ_API_KEY
PASO 5 → Docstrings código (Fases 3+4 del plan 29-abr)
PASO 6 → F10 Multi-usuario sobre SQLite/Postgres
PASO 7 → Tests e2e mínimos
PASO 8 → Features avanzadas (F11, F12, F13, F14, F15, F16)
```

### BBDD en DBeaver — plan concreto
La BBDD ya existe en SQLite (`src/db/models.py` tiene los modelos). El objetivo es:
1. Conectar DBeaver al archivo `.sqlite` del proyecto
2. Ver las tablas: `appointments`, `habits`, `habit_logs`, `user_config`
3. Verificar que los datos de prueba están bien guardados
4. Cuando migremos a Postgres (F10 multi-usuario), será trivial

### Arquitectura multicanal — nota para el futuro
El bot de Telegram es solo **una interfaz** sobre la API FastAPI. La arquitectura correcta es:
```
FastAPI (lógica central)
    ├── Bot Telegram (python-telegram-bot)  ← actual
    ├── WhatsApp (Twilio) ← futuro F14
    ├── Web app ← futuro
    └── Voz (Whisper) ← futuro F15
```
Esto significa que si la API está bien, añadir canales es fácil.
