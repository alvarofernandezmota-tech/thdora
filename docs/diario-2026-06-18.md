## Contexto

Necesitaba levantar el stack THDORA en una máquina diferente a la habitual.
El repo existía en `~/dev/thdora` en otra máquina pero en `madre` había que clonarlo.
Se encadenaron 4 bugs hasta conseguir que el bot arrancara.

---

## Bug 1 — `ImportError: cannot import name '_invalidate_cache'`

**Archivo:** `src/bot/handlers/nlp_disambig.py:25`

```
from src.bot.handlers.nlp import _invalidate_cache, _build_day_schedule, _time_to_min
ImportError: cannot import name '_invalidate_cache' from 'src.bot.handlers.nlp'
```

**Causa raíz:** La imagen Docker que corría el contenedor estaba desactualizada.
Fue construida antes de que el commit `83dfa5b` añadiera `_invalidate_cache`,
`_time_to_min` y `_build_day_schedule` a `nlp.py`.

**Lo que aprendí:** Cuando aparece `ImportError: cannot import name X`,
el problema es de **código** — o la función no existe, o la imagen está desactualizada.
Solución: `git reset --hard origin/main` + `docker compose build bot`.

---

## Bug 2 — `fatal: not a git repository` + `no configuration file provided`

**Síntoma:** Cualquier `git fetch` o `docker compose` fallaba.

**Causa raíz:** Ejecutaba los comandos desde `~` (home), no desde la carpeta del repo.
El repo además no estaba clonado en esa máquina — solo existía en `~/dev/thdora`
de otro equipo.

**Fix:**
```bash
cd ~/Projects
git clone git@github.com:alvarofernandezmota-tech/thdora.git
cd thdora
# ahora git y docker compose funcionan
```

**Lo que aprendí:** `git` y `docker compose` necesitan estar en la carpeta del proyecto
(donde está `.git` y `docker-compose.yml`). Si el `find . -maxdepth 3 -type d -iname "thdora"`
nο devuelve nada, hay que clonar de nuevo.

## Bug 3 — `.env not found`

**Síntoma:**
```
env file /home/varopc/Projects/thdora/.env not found
```

**Causa raíz:** Docker Compose busca `.env` en la misma carpeta donde se ejecuta.
El clone nuevo en `~/Projects/thdora` no tenía `.env` — estaba solo en el clone
antiguo de otra máquina.

**Fix:**
```bash
cp ~/dev/thdora/.env ~/Projects/thdora/.env
# El .env ya tiene todos los tokens configurados, no hay que recrearlo
```

**Lo que aprendí:** El `.env` nunca va al repo (está en `.gitignore`), así que
cada clone nuevo necesita su propio `.env`. Por eso existe `.env.example` —
es la plantilla para no tener que adivinar qué variables se necesitan.

## Bug 4 — `ModuleNotFoundError: No module named 'langgraph.checkpoint.sqlite'`

**Archivo:** `src/agents/memory/manager.py:30`

```
from langgraph.checkpoint.sqlite import SqliteSaver
ModuleNotFoundError: No module named 'langgraph.checkpoint.sqlite'
```

**Causa raíz:** En algún momento de la evolución de LangGraph, el módulo
`checkpoint.sqlite` se separó al paquete independiente `langgraph-checkpoint-sqlite`.
El `requirements.txt` solo tenía `langgraph>=0.2.0`, sin el paquete extra.

**Fix:** Añadido `langgraph-checkpoint-sqlite>=2.0.0` en `requirements.txt` y
`pyproject.toml`. Commit `bdf82f0` directo desde herramienta GitHub.
Después: `docker compose build bot` en la máquina.

**Lo que aprendí:** Cuando el error es `ModuleNotFoundError: No module named X`,
el problema es de **dependencias** (librerías), no de código interno.
Hay que buscar en `requirements.txt` / `pyproject.toml` y añadir el paquete que falta.

---

## Diferencia clave ImportError vs ModuleNotFoundError

| Error | Capa | Dónde mirar |
|-------|------|-------------|
| `ImportError: cannot import name X` | Código interno | El archivo `.py` que define `X` |
| `ModuleNotFoundError: No module named X` | Dependencias | `requirements.txt` / `pyproject.toml` |
